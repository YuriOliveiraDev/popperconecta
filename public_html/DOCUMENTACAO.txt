1. VISÃO GERAL
================================================================================

Popper Conecta é uma intranet corporativa e plataforma de business intelligence
para a empresa Popper. Desenvolvida em PHP (sem framework), MySQL e JavaScript
vanilla, integra com sistemas ERP (TOTVS), RH (Solides), workflows (Pipefy) e
e-mail corporativo (Microsoft Graph).

URL de produção: https://popperconecta.com.br
Ambiente atual:  prod (definido em app/config/config.php)


================================================================================
2. ARQUITETURA
================================================================================

Padrão:
- PHP tradicional (sem framework MVC)
- Cada página carrega bootstrap.php que inicializa sessão, DB e helpers
- Autenticação via sessão PHP + remember-me por cookie
- APIs internas retornam JSON
- Frontend: HTML + CSS customizado + JavaScript vanilla (sem React/Vue)
- Fontes externas: Bootstrap, Chart.js, Leaflet, Select2, Font Awesome

Fluxo de carregamento de uma página:
1. Página PHP chama  require_once bootstrap.php
2. bootstrap.php carrega config, DB, helpers, autenticação
3. Página chama require_login() ou require_admin() para verificar acesso
4. Página renderiza HTML com dados do DB e de APIs externas
5. JavaScript do cliente busca dados adicionais via fetch() em /api/

6. AUTENTICAÇÃO E AUTORIZAÇÃO
================================================================================

--- Autenticação ---

Fluxo de login (app/core/auth.php):
1. POST /login.php → login($email, $pass, $remember)
2. Busca usuário ativo por email
3. Valida senha com password_verify()
4. Regenera session_id()
5. Armazena usuário na sessão ($_SESSION['user'])
6. Se remember=true: gera token selector:validator, salva hash na tabela
   user_remember_tokens com expiração de 30 dias

Cookie de sessão: POPPERSESSID (prod) / POPPERSESSID_DEV (dev)
  - HttpOnly, SameSite=Lax, Secure (em HTTPS)

Remember-me:
  - Cookie remember_me = selector:validator (texto plano)
  - DB guarda hash(validator) — pattern industry-standard
  - Rotacionado a cada uso (previne session fixation)
  - Expira em 30 dias

Recuperação de senha:
  1. POST /esqueci-senha.php → gera token de 32 bytes, armazena hash + expiry 30min
  2. Envia link por e-mail via Microsoft Graph
  3. GET /reset-password.php?token=... → valida e permite nova senha

--- Autorização ---

Roles:
  admin   Acesso total
  (outro) Acesso por permissões granulares

Permissões (app/core/permissions.php):
  admin.users           Gestão de usuários
  admin.comunicados     Gestão de comunicados
  admin.rh              Módulo de RH
  admin.metrics         Configuração de métricas
  dash.comercial.executivo    Dashboard executivo
  dash.comercial.faturamento  Dashboard de faturamento
  dash.comercial.clientes     Dashboard de clientes
  dash.financeiro.inadimplencia    Dashboard de inadimplência
  dash.financeiro.contasp     Dashboard contas a pagar
  dash.comex.importacoes      Dashboard de importações

Funções de verificação:
  require_login()             Redireciona para /login.php se não autenticado
  require_admin()             Exige role=admin
  require_admin_perm($perm)   Exige admin + permissão específica
  require_dash_perm($perm)    Exige permissão de dashboard
  user_can($perm, $user)      Retorna bool — verifica permissão

Bloqueio de mobile:
  - Dispositivos móveis redirecionados para tela de bloqueio
  - Exceções: home, coins, dashboards, admin (mobile_shell_enabled)
  - Controlado em app/core/auth.php via User-Agent

10. SISTEMA DE POPPER COINS
================================================================================

O sistema de fidelidade interno permite que colaboradores acumulem e resgatem
coins por participar de campanhas e atividades.

Componentes:
  app/services/poppers_coins.php   Lógica principal
  coins/coins.php                  Saldo e extrato do usuário
  coins/coins_campanhas.php        Participação em campanhas
  coins/coins_resgatar.php         Solicitação de resgate
  coins/ranking.php                Ranking de colaboradores
  admin/rh/rh_coins.php            Gestão admin de saldo
  admin/rh/rh_rewards.php          Catálogo de recompensas
  admin/rh/rh_campaigns.php        Criação de campanhas
  admin/rh/rh_redemptions.php      Aprovação de resgates

Fluxo de resgate:
  1. Usuário acessa /coins/coins_resgatar.php
  2. Escolhe recompensa do catálogo
  3. Submete com token anti-duplicata (armazenado em sessão)
  4. Sistema cria registro em popper_coin_redemptions (status: pendente)
  5. Cria card no Pipefy via GraphQL para aprovação
  6. Admin aprova em /admin/rh/rh_redemptions.php
  7. Saldo debitado via apply_ledger() na carteira do usuário
  8. Usuário notificado via sistema de notificações

12. CACHE
================================================================================

Três níveis de cache:

1. localStorage (cliente)
   - KPIs do hero da home
   - Chave: index2:hero-kpis:v1
   - TTL: 5 minutos
   - Fallback: usa cache expirado se API falhar

2. Cache em arquivo /tmp/ (servidor)
   - Dados TOTVS (dashboards executivo, faturamento, etc.)
   - TTL: 2 minutos
   - Padrão de nome: /tmp/totvs_exec_*.json
   - Forçar refresh: ?force=1 na query string

3. Cache em arquivo JSON (servidor, persistente)
   - Catálogo de fornecedores: app/cache/fornecedores.json
   - Dados COMEX: cache/comex-importacoes.json
   - Rankings mensais: dashboards/cache/dash_rankings_*.json

13. FLUXOS PRINCIPAIS
================================================================================

--- Carregamento dos KPIs da Home ---
1. Usuário acessa /index.php (autenticado)
2. PHP renderiza página com placeholders "R$ --"
3. JS lê localStorage (index2:hero-kpis:v1) — se fresco (< 5 min), exibe
4. Se cache expirado: fetch GET /api/dashboard/dashboard-data.php?dash=executivo
5. API verifica cache /tmp/ (2 min) ou consulta TOTVS via TotvsAgentService
6. Retorna { values: {mes_total, meta_mes, atingimento_mes_pct, hoje_total}, updated_at }
7. JS formata como moeda BRL e atualiza DOM
8. Armazena no localStorage

--- Fluxo de Recuperação de Senha ---
1. Usuário acessa /esqueci-senha.php, digita e-mail
2. Sistema verifica se usuário existe e está ativo
3. Gera 32 bytes aleatórios → token em texto claro, hash armazenado
4. Link: https://popperconecta.com.br/reset-password.php?token=TOKEN
5. E-mail enviado via Microsoft Graph (remetente: no-reply@popper.com.br)
6. Usuário clica no link dentro de 30 minutos
7. /reset-password.php valida token (hash_equals, não expirado, não usado)
8. Usuário define nova senha → password_hash() armazenado
9. Token marcado como usado (used_at)

--- Upload do Popper News ---
1. Admin acessa /admin/popper_news.php
2. Faz upload de PDF ou imagens da edição
3. Arquivos salvos em /uploads/popper-news/current/
4. Home (/index.php) lê automaticamente as imagens dessa pasta
5. Exibe carrossel com autoplay de 4,2 segundos

--- Resgate de Popper Coins ---
1. Usuário acessa /coins/coins_resgatar.php
2. Visualiza catálogo (popper_coin_rewards)
3. Verifica saldo em popper_coin_wallets
4. Submete resgate com token anti-duplicata
5. Registro criado em popper_coin_redemptions (status: pendente)
6. Card criado no Pipefy via GraphQL
7. Admin aprova em /admin/rh/rh_redemptions.php
8. Coins debitados da carteira via ledger
9. Notificação enviada ao usuário

14. SEGURANÇA
================================================================================

Práticas implementadas:
  - Prepared statements PDO em todas as queries (SQL injection prevenido)
  - htmlspecialchars(ENT_QUOTES, UTF-8) em toda saída HTML (XSS prevenido)
  - password_hash(PASSWORD_DEFAULT) para senhas
  - Session ID regenerado no login (session fixation prevenido)
  - Cookies HttpOnly + SameSite=Lax + Secure (em HTTPS)
  - Token de recuperação de senha com validade de 30 minutos
  - Token remember-me com hash no banco (selector:validator pattern)
  - display_errors desabilitado em produção
  - Controle de acesso por role + permissões granulares
  - Token interno (POPPER_INTERNAL_AGENT_TOKEN) para chamadas entre serviços

Pontos de atenção para próximas versões:
  - Verificação SSL do TOTVS desabilitada (TOTVS_DISABLE_SSL = true)
    Risco: MITM. Solução: habilitar SSL no endpoint TOTVS ou usar gateway
  - Ausência de rate limiting no login (risco de brute force)
    Solução: implementar throttle por IP / e-mail
  - Ausência de tokens CSRF nos formulários admin
    Risco: CSRF em ações de escrita
  - Webhook do Solides sem validação de assinatura
    Solução: validar HMAC/header-secret enviado pelo Solides
